АНОТАЦІЯ
Кваліфікаційна робота присвячена проєктуванню та розробці сучасного
клієнт-серверного вебзастосунку для зручного онлайн-бронювання авіаквитків
та помешкань з інтеграцією платіжних рішень.
У першому розділі проведено аналіз предметної області, досліджено
існуючі системи онлайн-бронювання та визначено функціональні й
нефункціональні вимоги до програмного забезпечення. На основі аналізу
обґрунтовано вибір сучасного технологічного стека для реалізації проєкту.
У другому розділі виконано проєктування базової архітектури
вебзастосунку та структури програмних інтерфейсів. Розроблено концептуальну
та фізичну моделі реляційної бази даних, а також спроєктовано підсистему
комплексної безпеки, яка включає криптографічний захист даних та рольову
модель розмежування доступу.
У третьому розділі здійснено комплексну програмну реалізацію
клієнт-серверного вебзастосунку: створено користувацький інтерфейс із
логікою багатокритеріального пошуку та транзакційним алгоритмом
бронювання, а також розроблено серверну інфраструктуру (REST API) із
налаштуванням безпечної взаємодії з базою даних, генерацією документів і
панеллю адміністрування.
У четвертому розділі розглянуто методи комплексного тестування
розробленого програмного забезпечення. Спроєктовано конвеєр безперервної
інтеграції та доставки (CI/CD) і виконано успішне розгортання програмного
комплексу в хмарному середовищі з оцінкою його продуктивності.
КЛЮЧОВІ СЛОВА: FULL-STACK РОЗРОБКА, ВЕБЗАСТОСУНОК,
REST API, ORM-ВІДОБРАЖЕННЯ, КРИПТОГРАФІЧНИЙ ЗАХИСТ, ХМАРНЕ
СЕРЕДОВИЩЕ, АВТОМАТИЗАЦІЯ РОЗГОРТАННЯ.

SUMMARY
The qualification work is devoted to the design and development of a modern
client-server web application for convenient online booking of flight tickets and
accommodations with the integration of payment solutions.
In the first section, an analysis of the subject area is conducted, existing online
booking systems are investigated, and functional and non-functional requirements for
the software are determined. Based on the analysis, the choice of a modern
technology stack for the project implementation is justified.
In the second section, the design of the basic architecture of the web
application and the structure of software interfaces is carried out. Conceptual and
physical models of the relational database are developed, and a comprehensive
security subsystem is designed, which includes cryptographic data protection and a
role-based access control (RBAC) model.
In the third section, a comprehensive software implementation of the
client-server web application is carried out: a user interface with multi-criteria search
logic and a transactional booking algorithm is created, and the server infrastructure
(REST API) is developed, including the configuration of secure database interaction,
document generation, and an administration panel.
In the fourth section, methods of comprehensive testing of the developed
software are considered. A continuous integration and continuous delivery (CI/CD)
pipeline is designed, and the successful deployment of the software complex in a
cloud environment with an assessment of its performance is carried out.
KEY WORDS: FULL-STACK DEVELOPMENT, WEB APPLICATION,
REST API, ORM MAPPING, CRYPTOGRAPHIC PROTECTION, CLOUD
ENVIRONMENT, DEPLOYMENT AUTOMATION.

ВСТУП
Актуальність теми зумовлена необхідністю подолання розрізненості
сучасних туристичних сервісів: користувачі потребують єдиної екосистеми для
синхронного бронювання авіаквитків та готелів, що мінімізує часові витрати і
забезпечує релевантність пропозицій. Створення вебзастосунку, який об'єднує
ці функції з надійним захистом даних, є критично важливим для підвищення
якості обслуговування в індустрії туризму.
Мета і завдання дослідження. Метою роботи є проєктування та
програмна реалізація вебзастосунку для онлайн-купівлі авіаквитків та
бронювання помешкань.
Об’єктом дослідження є процеси автоматизації туристичних послуг у
мережі Інтернет.
Предметом дослідження виступають методи, алгоритми та програмні
засоби розробки вебсистем із використанням React, Node.js та PostgreSQL.
Методи дослідження. У роботі використано методи системного та
порівняльного аналізу, об'єктно-орієнтованого моделювання, ітеративного
прототипування та функціонального тестування.
Наукова новизна одержаних результатів полягає у вдосконаленні
архітектурних підходів до побудови систем бронювання шляхом поєднання
модулів пошуку авіарейсів та готелів у єдиному інтерфейсному рішенні з
використанням сучасних засобів управління станом та безпечної взаємодії з
даними через ORM-системи.
Практичне значення роботи визначається створенням готового до
експлуатації програмного продукту, який може бути використаний як база для
комерційних тревел-платформ або внутрішніх корпоративних систем
управління відрядженнями.
Структура роботи. Кваліфікаційна робота складається з чотирьох
основних розділів. Загальний обсяг основної частини становить 86 сторінок. До
роботи включено список використаних джерел, який налічує 21 позицій.

1.1 Аналіз предметної області та дослідження існуючих методів
організації систем онлайн-бронювання
Сучасний ринок туристичних послуг зазнав докорінних змін із
впровадженням концепції Travel Tech, яка перетворила традиційне бронювання
на складний технологічний процес. Предметна область дослідження охоплює
взаємодію між постачальниками послуг (авіакомпаніями, готелями) та
кінцевими споживачами через цифрові платформи. Головною особливістю цієї
сфери є критична залежність від актуальності даних: ціна та наявність місць
можуть змінюватися щосекунди, що ставить високі вимоги до швидкодії та
архітектурної стійкості програмних рішень [1].
Аналізуючи існуючі методи організації систем, можна виділити два
домінуючі підходи. Перший - це використання Глобальних дистриб'юторських
систем (GDS), таких як Amadeus або Sabre, які виступають агрегаторами для
тисяч постачальників. Другий підхід базується на прямій інтеграції з API
конкретних компаній або використанні метапошукових алгоритмів. Попри
ефективність світових лідерів (наприклад, Booking.com чи Skyscanner),
користувачі часто стикаються з проблемою фрагментації: для повного
планування подорожі доводиться перемикатися між різними сервісами, що
створює розрив у користувацькому досвіді.
Проведений аналіз (див. Таблицю 1.1) демонструє, що ключовою
проблемою існуючих великих платформ є архітектурна розрізненість.

посередником, не забезпечуючи повний цикл транзакції всередині своєї
системи. Booking.com, хоч і пропонує широкий спектр послуг, використовує

реалізації архітектури та бази даних. Згідно з міжнародними стандартами
інженерії програмного забезпечення, вимоги поділяються на функціональні, що
визначають поведінку системи, та нефункціональні, що описують атрибути
якості та технічні обмеження [3].

користувачів в таблитці, що наведено в додатку А. Це дозволяє реалізувати
гнучку модель розмежування доступу (RBAC), що є критично важливим для
безпеки транзакційних даних.

1.3 Порівняльний аналіз та вибір технологічного стека для реалізації
клієнт-серверної архітектури
Вибір технологічного стека є стратегічним етапом проєктування, що
визначає продуктивність, масштабованість та швидкість розробки
інформаційної системи. Для реалізації вебзастосунку було обрано
клієнт-серверний архітектурний патерн, який забезпечує чітке розділення
логіки представлення та обробки даних.
На рівні фронтенд-розробки розглядався вибір між бібліотекою React.js та
фреймворком Next.js. Попри популярність стандартного React, для системи
бронювання критично важливими є показники SEO та ключова метрика LCP.
Фреймворк Next.js пропонує механізми SSR та SSG, що дозволяє значно
пришвидшити відображення результатів пошуку та покращити індексацію
сторінок пошуковими системами (див. Рисунок 1.2).

Вибір зумовлений подійно-орієнтованою, неблокуючою моделлю
введення-виведення, яка ідеально підходить для обробки великої кількості

одночасних запитів, що характерно для систем бронювання [5]. У поєднанні з
TS, це забезпечує сувору статичну типізацію даних, мінімізуючи кількість
логічних помилок, що значно підвищує надійність та підтримуваність кодової

Для управління даними було обрано реляційну СУБД PostgreSQL. На

принципам ACID, що є безальтернативною вимогою для фінансових транзакцій
та збереження цілісності бронювань [6]. Взаємодія між сервером та базою
даних реалізована за допомогою Prisma ORM. Використання ORM-системи
нового покоління дозволяє автоматизувати міграції, забезпечити типобезпечні
запити до БД та надати розробнику потужний інструментарій для маніпуляції
складними зв'язками між сутностями (наприклад, зв'язок між користувачем,
рейсом та готелем). Така комбінація реляційної СУБД із ORM-прошарком
нового покоління забезпечує як надійність зберігання транзакційних даних, так
і гнучкість у роботі зі складними зв'язками між сутностями на рівні застосунку.
Фінальним етапом вибору стека є визначення інфраструктури для
розгортання та забезпечення CI/CD. Для розгортання обрано хмарну платформу
Vercel із вбудованою Git Integration, яка забезпечує автоматичну збірку,
тестування та деплой при кожному push у репозиторій без необхідності окремої
CI-інфраструктури. Платформа Vercel є ідеальним рішенням для
Next.js-додатків, оскільки пропонує нативну підтримку SSR, автоматичну
оптимізацію CDN та безшовне розгортання «серверлес»-функцій для
API-ендпоінтів. Додатковою перевагою є автоматичне створення Preview
Deployment для кожної гілки репозиторію, що дозволяє перевіряти зміни в
ізольованому середовищі до їх злиття у головну гілку, що забезпечує високу
швидкість розробки та стабільність продуктивного середовища [7] (див.
Рисунок 1.3). Конвеєр збірки включає послідовні етапи статичного аналізу коду,
запуску unit-тестів та перевірки типів TypeScript, що унеможливлює
потрапляння некоректного коду у продуктивне середовище.

РОЗДІЛ 2. ПРОЄКТУВАННЯ АРХІТЕКТУРИ ТА БАЗИ ДАНИХ
2.1 Проєктування базової архітектури вебзастосунку
Процес побудови архітектури системи онлайн-бронювання базується на
принципах декомпозиції складних бізнес-процесів на окремі функціональні
модулі. Основною метою цього етапу є створення гнучкої структури, яка
забезпечить стабільну взаємодію між клієнтською частиною та сервером бази
даних, мінімізуючи затримки при обробці запитів.
В основу архітектурного рішення покладено сучасні патерни розробки,
що дозволяють ізолювати рівень представлення даних від логіки їх збереження
та обробки. Це створює умови для незалежного тестування компонентів та
полегшує подальше масштабування системи при зростанні кількості
користувачів.
2.1.1 Обґрунтування клієнт-серверної моделі взаємодії та вибір
архітектурних патернів
Для реалізації вебзастосунку обрано трирівневу клієнт-серверну
архітектуру (Three-Tier Architecture) (див. Рисунок 2.1), яка є стандартом для
сучасних високонавантажених систем. Дана модель передбачає поділ
застосунку на три незалежні рівні: рівень представлення (Frontend), рівень
бізнес-логіки (Backend) та рівень даних (Database). Така структура дозволяє
чітко розмежувати відповідальність між компонентами: клієнтська частина
відповідає лише за інтерфейс та обробку подій користувача, тоді як серверна
частина бере на себе складні обчислення, валідацію та безпеку [8].
Незалежність рівнів також забезпечує можливість горизонтального
--- PAGE 22 ---

масштабування кожного компонента окремо, що є критично важливим в умовах
пікового навантаження, характерного для систем бронювання.

2.1.2 Проєктування структури програмних інтерфейсів та
маршрутизації даних
Взаємодія між фронтенд-архітектурою FSD та сервером базується на
принципах REST, що дозволяє уніфікувати доступ до ресурсів системи через
стандартні HTTP-методи. Проєктування програмних інтерфейсів у системі
спрямоване на забезпечення високої швидкодії, безпеки та зрозумілості
структури запитів.
Центральним вузлом маршрутизації є файл router/index.js, який виконує
роль головного диспетчера запитів. Для забезпечення чистоти коду та логічного
розділення доступу маршрути групуються за функціональним призначенням.
Повний перелік маршрутів із зазначенням методів, ендпоінтів та рівнів доступу
наведено в таблиці в Додатку Б.
Ключовою особливістю розробленої маршрутизації є використання
ланцюжка проміжного програмного забезпечення. Кожен запит проходить через
етапи валідації та автентифікації перед потраплянням до бізнес-логіки. На
прикладі реалізованого модуля автентифікації структура роутів виглядає
наступним чином:
ー реєстрація та вхід: запити /registration та /login використовують
схему валідації для перевірки коректності введених даних на рівні HTTP-запиту,
що запобігає передачі некоректних даних у сервісний шар;
ー керування сесіями: маршрути /logout та /refresh відповідають за
життєвий цикл токенів Access та Refresh, реалізуючи механізм безпечного
оновлення сесії без повторного введення облікових даних;
ー активація профілю: ендпоінт /activate/:link демонструє використання
параметризованих маршрутів для підтвердження електронної пошти
користувача.
Важливою особливістю проєктування є розмежування рівнів доступу на
етапі маршрутизації. Використання authMiddleware дозволяє автоматично

2.2.2 Фізичне проєктування реляційної структури засобами
PostgreSQL та налаштування схем Prisma ORM
Фізичне проєктування полягає у трансформації концептуальної моделі у
конкретну реляційну схему бази даних СУБД PostgreSQL. Для реалізації цього
етапу використано Prisma ORM, яка дозволяє описувати структуру за
допомогою декларативної мови Prisma Schema Language. Такий підхід
автоматизує генерацію SQL-міграцій та забезпечує повну наскрізну типізацію
даних у програмному коді.
Як базовий приклад фізичної реалізації сутності розглянемо модель User,
що є центральним вузлом системи автентифікації та профілювання
користувачів, реалізацію якої наведено у Додатку В.
Модель User поєднує механізми автентифікації, рольового доступу та
зв'язки з транзакційними модулями системи. Використання типу UUID як
первинного ключа унеможливлює вгадування ідентифікаторів через
послідовний перебір, що є критичним для систем із фінансовими операціями.
Поля email та activationLink мають обмеження @unique, що гарантує цілісність
даних на рівні СУБД та запобігає дублюванню облікових записів. Рольова
модель реалізована через enum Role зі значеннями GUEST, CLIENT та ADMIN,
що забезпечує гнучке розмежування доступу без окремої таблиці прав. Логіка
активації побудована на парі полів isActivated та activationLink – після
підтвердження електронної пошти прапорець переходить у true, а посилання
анулюється. Директиви @default(now()) та @updatedAt автоматизують аудит
часу реєстрації та останньої активності без додаткового коду на рівні бекенду.
Зв'язки Token[] та Booking[] демонструють готовність архітектури до підтримки
багатьох одночасних сесій з різних пристроїв та повної історії бронювань у
межах одного облікового запису.
Решта сутностей схеми реалізують предметну логіку системи. Моделі
Airport та Airline зберігають довідникові дані авіаційної інфраструктури з
--- PAGE 30 ---
30
IATA-кодами як унікальними ідентифікаторами. Модель Flight пов'язує обидва
аеропорти через два окремі зовнішні ключі – departureAirportId та
arrivalAirportId – і містить поле availableSeats для контролю доступності місць у
реальному часі; складений індекс на (departureAirportId, arrivalAirportId,

зручності зберігаються як масиви String[], а унікальний констрейнт [hotelId,
roomNumber] виключає дублювання номерів у межах одного готелю. Сутність
Passenger винесена окремо від бронювання для зберігання паспортних даних
без захаращення транзакційної моделі.
Центральною транзакційною сутністю є модель Booking, яка через поле
type (enum BookingType) визначає тип бронювання та утворює відносини «один
до одного» з відповідною дочірньою моделлю – FlightBooking або HotelBooking.
Такий підхід дозволяє зберігати уніфікований статус оплати та посилання на
Stripe Payment Intent на рівні батьківської сутності, забезпечуючи атомарність
транзакції незалежно від типу послуги. Модель Payment фіксує повний
аудиторський слід платежу: ідентифікатори Stripe, статус, а також часові мітки
paidAt та refundedAt для відстеження повернень коштів.
Таким чином, фізичне проєктування засобами PostgreSQL та Prisma ORM
дозволило створити надійний рівень збереження даних із чіткими реляційними
зв'язками, вбудованою системою міграцій та індексацією критичних полів
пошуку, що є необхідною умовою для стабільної роботи вебзастосунку.
2.3 Проєктування підсистеми комплексної безпеки вебзастосунку
Забезпечення конфіденційності, цілісності та доступності даних є
фундаментальним пріоритетом при розробці сервісів онлайн-бронювання, що
оперують персональною інформацією користувачів та забезпечують життєвий
цикл транзакцій. Проєктування підсистеми безпеки вебзастосунку базується на
--- PAGE 31 ---
31
принципі багатоешелонованого захисту, що передбачає створення незалежних
рівнів контролю на етапах передачі даних, автентифікації сесій та збереження
конфіденційних атрибутів у базі даних.
Головною метою розробленої підсистеми є мінімізація ризиків
компрометації облікових записів та захист від поширених вразливостей, таких
як XSS та CSRF. Архітектурне рішення підсистеми безпеки включає:
ー криптографічний захист: використання адаптивних алгоритмів
хешування для незворотного перетворення паролів;
ー двофакторну логіку доступу: поєднання стандартної автентифікації
з механізмом верифікації через зовнішні поштові сервіси;
ー stateless-авторизацію: впровадження стандарту JSON Web Token для
розмежування прав доступу без навантаження на оперативну пам'ять сервера;
ー ізоляцію середовища: застосування механізмів httpOnly та Secure
для захисту токенів на стороні клієнта.
Такий комплексний підхід дозволяє створити стійке до зовнішніх загроз
середовище, де безпека не є ізольованим модулем, а інтегрована безпосередньо
в бізнес-логіку та маршрутизацію застосунку. Це гарантує надійність системи
як для кінцевих користувачів, так і для адміністраторів платформи.
2.3.1 Моделювання процесів ідентифікації, автентифікації та
криптографічного захисту даних
Процес надання користувачеві доступу до захищених ресурсів системи
декомпозовано на три логічні етапи: ідентифікацію (визначення користувача за
унікальним email), автентифікацію (підтвердження особи за допомогою пароля)
та авторизацію (надання прав доступу на основі згенерованих токенів). Для
забезпечення максимального рівня безпеки на кожному з цих етапів
застосовуються спеціалізовані криптографічні алгоритми та протоколи.
--- PAGE 32 ---
32
Зберігання паролів у базі даних у відкритому вигляді є критичною
вразливістю. У розробленій системі реалізовано незворотне хешування
автентифікаційних даних за допомогою алгоритму Bcrypt [11]. На відміну від
стандартних криптографічних хеш-функцій (наприклад, SHA-256), Bcrypt є
обчислювально складним алгоритмом, який використовує механізм генерування
унікальної «солі» для кожного запису та фактор складності (cost factor).
Математично процес створення захищеного запису при реєстрації можна
описати як:
Hash = Bcrypt(Password, Salt, Cost) (2.1)
де:
ー Hash – криптографічний хеш, що зберігається в базі даних (поле
password у таблиці User);
ー Password – пароль у відкритому вигляді, введений користувачем при
реєстрації;
ー Salt – унікальна випадкова послідовність символів ("сіль"),
згенерована алгоритмом для конкретного запису;
ー Cost – фактор обчислювальної складності (кількість ітерацій
хешування, що визначає час обчислення).
Додавання солі унеможливлює злом паролів за допомогою заздалегідь
обчислених райдужних таблиць, а фактор складності (у проєкті встановлено
значення cost factor рівне 10, що відповідає загальноприйнятим рекомендаціям
OWASP та забезпечує час обчислення близько 100–150 мс на типовому
виробничому сервері – достатньо для унеможливлення масових атак перебором,
але не відчутно для кінцевого користувача) робить атаки методом повного
перебору економічно недоцільними для зловмисника. У базі даних, у таблиці
User зберігається виключно фінальний результат роботи алгоритму.

Для керування станом авторизованого користувача замість класичних
серверних сесій обрано архітектуру на базі токенів (Stateless Authentication). Це
рішення ідеально інтегрується з клієнтським FSD-застосунком та Redux Toolkit,
оскільки сервер не зберігає стан підключення в оперативній пам'яті, що суттєво
підвищує масштабованість API.
Після успішної перевірки хешу пароля сервіс token-service.js генерує пару
токенів за стандартом RFC 7519 [10]:
ー access token: короткостроковий токен (із терміном життя 15–30
хвилин), що містить у своєму корисному навантаженні ідентифікатор
користувача, його email та статус активації. Токен підписується секретним
ключем сервера (HMAC SHA-256). Він передається клієнту в тілі відповіді,
зберігається в Redux Store і додається до кожного запиту в заголовку
Authorization: Bearer <token>;
ー refresh token: Довгостроковий токен (із терміном життя до 30 днів).
Його головна мета - непомітне для користувача отримання нового Access-токена
після закінчення терміну його дії.
Для мінімізації ризиків викрадення токенів через атаки типу XSS, Refresh
Token передається клієнту не в тілі JSON-відповіді, а встановлюється через
заголовок Set-Cookie з прапорцем httpOnly. Це робить токен недоступним для
читання через JavaScript (наприклад, через document.cookie).
Коли клієнтський застосунок фіксує закінчення дії Access-токена або його
відсутність (наприклад, після перезавантаження сторінки), він автоматично
звертається до ендпоінта /api/refresh. Сервер зчитує httpOnly куку, перевіряє
наявність цього Refresh-токена у таблиці Token бази даних і, у разі валідності,
проводить процедуру ротації: старий токен видаляється, а нова пара (Access +
Refresh) повертається клієнту. Це гарантує безперервний користувацький досвід
та безпеку сесії.
Фінальним ешелоном захисту є auth-middleware.js. Перед тим, як
будь-який запит до захищеного маршруту (наприклад, /api/booking) потрапить

до контролера, він проходить через цей шар. Middleware виконує парсинг
заголовка Authorization, витягує Access Token і криптографічно перевіряє його
підпис за допомогою секретного ключа (JWT_ACCESS_SECRET). Якщо підпис
не збігається, токен прострочений або відсутній, система негайно перериває
обробку запиту та повертає стандартизовану помилку доступу (HTTP Status 401
Unauthorized), запобігаючи несанкціонованому виконанню бізнес-логіки.
Програмну реалізацію описаних алгоритмів винесено в окремі модулі.
Лістинг сервісу керування користувачами (user-service.js), що інкапсулює
бізнес-логіку реєстрації, хешування паролів та ротації токенів, наведено в
додатку Д. Програмний код проміжного програмного забезпечення
(auth-middleware.js), який відповідає за перехоплення запитів та верифікацію
підпису Access-токена, наведено в додатку Е.
2.3.2 Проєктування рольової моделі розмежування доступу
У багатокористувацьких системах онлайн-бронювання критично
важливою є проблема розмежування повноважень, щоб унеможливити доступ
звичайних клієнтів до інструментів управління контентом платформи. Для
вирішення цього завдання в системі спроєктовано модель керування доступом
на основі ролей RBAC. Даний підхід дозволяє призначати привілеї не кожному
користувачеві індивідуально, а прив'язувати їх до стандартизованих ролей.
Базова архітектура передбачає наявність двох основних рівнів привілеїв:
ー клієнт (CLIENT): базовий рівень. Користувач із цією роллю має
право виконувати пошук авіарейсів та готелів, створювати бронювання,
переглядати власну історію транзакцій та керувати персональним профілем;
ー адміністратор (ADMIN): розширений рівень. Надає повний доступ
до управління ресурсами системи (CRUD-операції над сутностями рейсів та
готелів), а також можливість перегляду глобальної статистики та управління
обліковими записами клієнтів.
--- PAGE 35 ---
35
На даталогічному рівні, у схемі бази даних Prisma, інтеграція RBAC
здійснюється шляхом розширення сутності User. До моделі додається поле role,
яке базується на перелічуваному типі даних (enum Role { CLIENT, ADMIN }).
За замовчуванням директива @default(CLIENT) гарантує, що під час
стандартної реєстрації новий профіль автоматично отримує базові права.
Інтеграція рольової моделі в підсистему безпеки тісно пов'язана з
механізмом JWT. Під час авторизації ідентифікатор ролі додається до payload
access-токена. Зокрема, до складу payload включаються поля id, email,
isActivated та role, що формують мінімально достатній контекст для прийняття
рішення про авторизацію без звернення до бази даних. Таким чином, перевірка
прав доступу відбувається виключно на рівні підпису токена та його вмісту, що
відповідає принципу stateless-автентифікації та суттєво знижує навантаження на
СУБД при масових запитах.
Для захисту адміністративних ендпоінтів реалізовано спеціалізоване
проміжне програмне забезпечення adminMiddleware, що застосовується у парі з
authMiddleware до всіх маршрутів групи /api/admin/*. Його алгоритм роботи
базується на принципі композиції з auth-middleware:
1. Система спочатку перевіряє криптографічну валідність токена.
2. Витягує роль із розшифрованого payload.
3. Порівнює наявну роль із масивом дозволених ролей для
конкретного маршруту.
4. У разі невідповідності перериває ланцюжок виконання та повертає
помилку доступу (HTTP Status 403 Forbidden).
На клієнтській стороні, архітектура FSD та стан Redux використовують
отриману роль для динамічного рендерингу інтерфейсу. Наприклад, віджети
панелі адміністратора або кнопки редагування рейсів повністю приховуються
для користувачів із роллю CLIENT. Такий комплексний підхід до реалізації
RBAC забезпечує високий рівень надійності та унеможливлює несанкціоновану
ескалацію привілеїв у системі.

2.3.3 Механізми захисту API та валідації потоків даних від поширених
мережевих вразливостей
Відкриті API є однією з головних мішеней для зловмисників. Згідно з
актуальними дослідженнями консорціуму OWASP, недостатня валідація даних,
ін'єкції та некоректна конфігурація безпеки є найпоширенішими причинами
компрометації систем [12]. Для захисту вебзастосунку на етапі маршрутизації та
обробки запитів впроваджено комплексний механізм фільтрації та безпеки
потоків даних.
Будь-які дані, що надходять від клієнта (тіло запиту req.body, параметри
URL req.params або запити req.query), вважаються апріорі небезпечними. Для
забезпечення їх цілісності в системі застосовано декларативний підхід до
валідації через проміжне програмне забезпечення validate(schema). До моменту
передачі даних у контролер, вони перевіряються на відповідність суворим
правилам. Якщо дані не валідні, запит негайно відхиляється.
Захист від SQL-ін'єкцій на рівні бази даних гарантується використанням
Prisma ORM. Усі запити до PostgreSQL генеруються абстрактним рушієм, який
автоматично використовує параметризовані SQL-запити. Цей механізм ізолює
логіку запиту від даних користувача, що математично унеможливлює виконання
вбудованого зловмисного коду [13].
У REST API часто виникає проблема надлишкового розкриття інформації,
коли база даних повертає більше полів, ніж потрібно клієнту. Для вирішення
цього завдання впроваджено архітектурний шаблон DTO.
Наприклад, клас UserDto виконує роль "фільтра вихідних потоків". Він
приймає повний об'єкт користувача з БД і формує новий об'єкт, що містить
лише публічно дозволені поля, свідомо ігноруючи хеш пароля та службові
часові мітки. Крім того, на рівні ORM суворо контролюється, які саме поля
дозволено оновлювати, що блокує атаки типу Mass Assignment [12].

37
Оскільки клієнтська частина та серверна частина розгорнуті як незалежні
вузли, для безпечної взаємодії між ними налаштовано політику CORS.
Конфігурація сервера базується на специфікаціях W3C та дозволяє приймати
запити виключно від авторизованого білого списку доменів, зазначених у
змінних оточення [14]. Усі запити з невідомих джерел блокуються на етапі
"preflight" (методом OPTIONS), що запобігає несанкціонованому використанню
API сторонніми скриптами.
Вразливістю багатьох систем є повернення клієнту технічних деталей
помилок, які можуть розкрити архітектуру сервера або структуру бази даних.
Для запобігання цьому розроблено глобальне проміжне програмне забезпечення
error-middleware.js, яке перехоплює всі виключні ситуації. У разі критичної
помилки на сервері, middleware логує технічні деталі у внутрішній файл, а
клієнту повертає стандартизований та знеособлений об'єкт помилки (HTTP 500
Internal Server Error), що не містить жодної чутливої інформації, усуваючи
загрозу Information Exposure [12].
Такий багаторівневий підхід до валідації потоків даних та конфігурації
API створює міцний бар'єр проти сучасних мережевих загроз, забезпечуючи
стабільну та безпечну роботу платформи бронювання.
Висновки до розділу 2
Проєктування архітектури та бази даних розкрило ключову технічну
закономірність системи онлайн-бронювання: жодне архітектурне рішення не
може розглядатись ізольовано. Вибір трирівневої моделі з чітким
розмежуванням Frontend, Backend та Database одразу визначив структуру
маршрутизації, форму передачі даних через DTO та спосіб організації
бізнес-логіки у патерні CSR. Методологія FSD на клієнті та централізоване
управління станом через Redux Toolkit сформували передбачувану архітектуру
інтерфейсу, де кожен шар має строго визначену зону відповідальності.
